# CP-0012 — Messages Side Channel v0

## Goal

Implement a minimal Messages Side Channel v0.

This task creates the long-lived architectural seam for campaign messages without building the full message system.

Messages are campaign records. They are part of the server-built campaign packet as a side channel, but they must not enter the main LLM narration context in v0.

Core rule:

```text
Messages are visible to the system by default.
Messages are visible to the LLM only by deliberate future controller selection.
```

## Product Model

This project is the Roleplaying Companion core product.

Do not frame this task around any external game client or setting-specific reference tool.

The relevant core product surfaces are:

```text
Prose:
The current generated scene, player action input, and immediate LLM turn loop.

Chronicle:
The committed campaign history and durable record of accepted events.

Mailbox:
A separate side-channel surface for campaign messages.
```

Messages belong to the Mailbox, not the Prose surface.

The Prose surface may show only a compact binary Mailbox availability indicator:

```text
Mailbox: Mail unavailable.
```

or:

```text
Mailbox: Mail available.
```

The Prose surface must not become the Mailbox.

The Chronicle must not be updated merely because messages were counted, checked, or displayed as available.

## Critical Architecture Rule

Messages must be part of the campaign packet as side-channel state:

```text
campaignPacket.sideChannels.messages
```

Messages must not be inserted into:

```text
campaignPacket.context
```

Messages must not be included in the LLM prompt as prose/narration context in v0.

Messages must not appear as message content inside the Prose surface.

Messages must not create Chronicle entries merely by being checked or shown as available.

## Current Example Message Shape

There is currently one message available in the messages folder with this intended structure:

```json
{
  "id": "msg_manteos_steward",
  "from": "Steward of Lord Manteos",
  "to": "player clan",
  "dueAfter": {
    "season": "Summer",
    "day": 1,
    "year": 1085,
    "block": "Morning"
  },
  "status": "delivered",
  "summary": "A steward reminds the clan that village recruitment under Manteos' seal requires permission or compensation.",
  "relatedThread": "thread_recruitment_rights",
  "deliveredAt": "Morning, Summer 1, 1085"
}
```

Use this example to infer structure only.

Do not hardcode this example, its ID, its sender, its thread, or its content into application logic.

## Required Responsibility Split

```text
Message side-channel controller validates messages and computes binary availability.
campaignPacketBuilder assembles the side-channel result.
Frontend renders only a compact binary Mailbox availability indicator.
LLM prompt/context path ignores messages in v0.
Chronicle is not affected by message availability checks.
```

`campaignPacketBuilder` must not become a message engine.

The frontend must not build or infer message state on its own.

## Minimal Message v0 Validation

A usable Message v0 record must have:

```text
id: string
from: string
to: string
dueAfter: object
dueAfter.season: string
dueAfter.day: number
dueAfter.year: number
dueAfter.block: string
status: string
summary: string
relatedThread: string
deliveredAt: string
```

For v0:

* do not enforce a full status enum
* do not enforce valid season/block enums unless a safe existing calendar utility already does this
* do not validate whether `relatedThread` exists
* do not interpret `relatedThread`
* do not validate sender/recipient as actor IDs
* do not create actor/message/thread links
* malformed records must be skipped
* malformed records must not crash the game loop

## Availability Rule v0

For v0, a message is available when:

```text
valid Message v0 record
AND status === "delivered"
AND dueAfter <= current campaign time
```

`status: "delivered"` means the message has reached the Mailbox.

It does not mean the message has been narrated in the Prose surface.

It does not mean the message has been read.

The output of the availability check must be binary:

```text
hasAvailable: true
```

or:

````text
 of the availability check must be binary:

```text
hasAvailable: true
````

or:

```text
hasAvailable: false
```

Do not expose the number of available messages in v0.

If safe campaign-time comparison is too risky or unclear after inspecting existing time utilities, implement the smallest safe comparison possible and report the limitation in `TECHNICAL_DEBT.md`.

Do not advance campaign time while checking message availability.

Do not mutate message status while checking message availability.

Do not create Chronicle entries while checking message availability.

## campaignPacket Requirement

Add a minimal binary side-channel summary to the server-built campaign packet.

Preferred shape:

```json
{
  "sideChannels": {
    "messages": {
      "hasAvailable": true
    }
  }
}
```

If `campaignPacket.sideChannels` already exists or project conventions require a slightly different shape, follow the existing convention and report the actual shape.

Adding a temporary `sideChannels.messages` summary to the server-built campaign packet is allowed. This does not count as campaign-state mutation.

Do not write this summary back into campaign state files.

Do not include message body text, summary text, sender, recipient, `relatedThread`, delivered timestamp, due date, or available message count in the packet side-channel v0.

For v0, the side channel is a binary presence summary only.

Skipped malformed message count may be logged and reported by Codex, but it must not be exposed as user-facing Mailbox state.

## LLM Context Rule
