# CP-0012b — Prevent Lenient Placeholder Records From Being Restored

## Goal

Patch a rollback/snapshot edge case introduced by lenient loading for messages.

Malformed message JSON may be loaded into runtime as a placeholder object so the app can skip it safely. That placeholder must never be written back to disk during snapshot restore.

This is a hotfix. Keep the change narrow.

## Current Issue

`store.js` now has a lenient JSON loader:

```js
function listJsonLenient(dirPath) {
  if (!fs.existsSync(dirPath)) return [];
  return fs.readdirSync(dirPath)
    .filter((name) => name.endsWith(".json"))
    .map((name) => {
      try {
        return readJson(path.join(dirPath, name));
      } catch (error) {
        return {
          __malformedJson: true,
          __fileName: name
        };
      }
    });
}
```

`loadCampaign()` currently uses it for:

```js
units: listJsonLenient(path.join(campaignDir, "units")),
messages: listJsonLenient(path.join(campaignDir, "messages")),
```

The risk is in snapshot/restore behavior:

```text
malformed message JSON
→ listJsonLentext
malient returns placeholder object
→ createSnapshot stores placeholder in snapshot campaign.messages
→ restoreSnapshot calls writeCollection("messages", campaign.messages)
→ writeCollection assumes record.id exists
→ possible messages/undefined.json write
```

This must be prevented.

## Critical Inspection Rule

Before implementing, inspect the actual current code.

Do not assume:

* `writeCollection()` is still shaped exactly as described here
* `restoreSnapshot()` writes all loaded collections
* lenient placeholder fields are exactly `__malformedJson` and `__fileName`
* message records are the only lenient records that can reach restore
* a generic write fix is safe for all collections

Patch against the actual implementation and report any correction.

## Required Behavior

Malformed lenient placeholder records may exist in loaded runtime campaign data for validation/skipping.

They must not be written back to disk during restore or any collection rewrite.

A malformed placeholder record is any record that is clearly not a persistable campaign record, especially:

```text
record.__malformedJson === true
```

or a record missing a valid non-empty string `id`.

## Preferred Fix Direction

Prefer a narrow helper that prevents non-persistable placeholder records from being written.

Possible safe direction:

```js
function isPersistableRecord(record) {
  return record &&
    record.__malformedJson !== true &&
    typeof record.id === "string" &&
    record.id.trim();
}
```

Then apply it only where safe.

Do not silently make all critical collections lenient unless that is clearly the smallest safe fix.

Hard-fail behavior for core non-lenient campaign files should remain intact unless the current code already behaves otherwise.

## Important Design Boundary

Lenient loading is allowed for optional/side-channel style collections like messages.

Lenient placeholder records are runtime diagnostic objects.

They are not campaign records.

They must not cross the persistence write boundary.

Industry rule for this patch:

```text
Runtime placeholders may be read.
Runtime placeholders may be skipped.
Runtime placeholders must not be written.
```
