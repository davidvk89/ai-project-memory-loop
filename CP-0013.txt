# CP-0013 — Deprecate Legacy Enforcement And Add Scoped Enforcement v0

## Status

Ready for Codex implementation.

## Development Pass

PASS 3 — Server-side Campaign Context Retrieval

## Objective

Deprecate the existing Bannerlord-shaped Enforcement model without deleting it, and introduce a clean core Enforcement v0 model for scoped laws, customs, obligations, and restrictions.

Enforcement v0 must establish the extension point for scoped normative campaign truth:

```text
Chronicle records what happened.
Enforcement records what must now be respected.
```

Clean Enforcement v0 records must support two separate surfaces:

* a clean Enforcement API/feed projection named `enforcementRecords`
* LLM prose context as binding constraints

These surfaces have different selection rules.

`enforcementRecords` is the second-Chronicle-style normative feed. It should contain all clean v0 Enforcement records, including active, resolved, and retired records, regardless of whether their scopes match the current active scope set.

LLM binding constraints are prose-control context. They should contain only clean v0 records that are `status: "active"` and whose typed scopes exactly match the current active scope set.

Frontend rendering may be added only if it is safe without broad frontend refactor.

## Core Product Definition

Enforcement records are scoped campaign consequences that future prose must respect when applicable.

They are not merely importance weights.

They are not just Chronicle events.

They are not automatically global.

They apply to prose only when their typed scopes exactly match the current active scope set.

## Current Legacy Situation

The current `campaign/enforcement/*.json` records are Bannerlord-shaped manual synchronization records.

Legacy examples include fields such as:

* `suggestedBannerlordAction`
* `bannerlordActionTaken`
* string-only `authority`

These are deprecated legacy/module-specific fields.

They must not define the new core Enforcement schema.

## Clean Enforcement v0 Schema

A clean Enforcement v0 record must match one of the accepted v0 record types.

### Law Example

```json
{
  "id": "enforce_wergild_law",
  "type": "law",
  "status": "active",
  "scopes": [
    {
      "type": "organization",
      "id": "org_al_sahir"
    }
  ],
  "statement": "Wergild is paid first to kin, then to battle-brothers if no kin exist. The clan takes no share.",
  "createdAt": "Summer 10, 1084"
}
```

### Custom Example

```json
{
  "id": "enforce_empty_cushion_custom",
  "type": "custom",
  "status": "active",
  "scopes": [
    {
      "type": "household",
      "id": "household_al_sahir"
    }
  ],
  "statement": "On the first night of each season, the household sets bread and drink for the absent and names one hope.",
  "createdAt": "Winter 2, 1085"
}
```

### Obligation Example

```json
{
  "id": "enforce_manteos_fine",
  "type": "obligation",
  "status": "resolved",
  "scopes": [
    {
      "type": "organization",
      "id": "org_player_clan"
    }
  ],
  "requiredAction": "Resolve Lord Manteos' demanded 5,000 denar fine.",
  "reason": "Manteos claims the clan recruited from villages under his protection without permission.",
  "resolution": "The player paid the fine and acknowledged his authority over recruitment in protected villages.",
  "createdAt": "Morning, Summer 1, 1085",
  "resolvedAt": "Midday, Summer 1, 1085"
}
```

## Required v0 Fields

Required for all clean v0 records:

* `id`: non-empty string
* `type`: non-empty string
* `status`: non-empty string
* `scopes`: non-empty array of typed scope references
* `createdAt`: non-empty string

Each scope must have:

* `type`: non-empty string
* `id`: non-empty string

Initial accepted `type` values:

* `law`
* `custom`
* `obligation`
* `restriction`

Initial accepted `status` values:

* `active`
* `resolved`
* `retired`

Use `retired` for clean in-world rules/customs that no longer apply.

Use “deprecated legacy” only for old Bannerlord-shaped records that do not match the clean v0 schema.

## Type-Specific Content Rules

For `type: "law"`:

* require `statement`
* do not require `requiredAction`

For `type: "custom"`:

* require `statement`
* do not require `requiredAction`

For `type: "restriction"`:

* require `statement`
* do not require `requiredAction`

For `type: "obligation"`:

* require `requiredAction`
* `reason` is optional but should be supported
* `resolution` and `resolvedAt` are optional and only meaningful for resolved obligations

Do not allow a law/custom/restriction to be considered clean v0 if it only has `requiredAction`.

Do not allow an obligation to be considered clean v0 if it lacks `requiredAction`.

Only `status: "active"` records may affect LLM prose context as binding constraints.

Resolved and retired records may be returned for the clean Enforcement API/feed projection, but they must not be treated as active constraints.

## Unknown Extra Fields Policy

Unknown extra fields on otherwise clean v0 records should not make the record invalid, unless they are known deprecated Bannerlord legacy fields such as:

* `suggestedBannerlordAction`
* `bannerlordActionTaken`

Unknown extra fields must not be inserted into LLM context.

Unknown extra fields must not be exposed as core UI/API fields.

This allows future fields such as `sourceEventId` without breaking v0, while still preventing deprecated Bannerlord fields from silently becoming core schema.

## Authority Modeling Is Deferred

Do not add:

* `authorityRef`
* authority hierarchy
* authority validation
* authority-to-scope matching
* issuer modeling
* legal precedence rules

Authority/source modeling is deferred to a later task.

Do not preserve legacy string `authority` as a clean v0 core field.

## Deprecated Legacy Record Rule

A record is deprecated legacy if it does not satisfy the clean v0 schema.

A record should also be treated as deprecated legacy if it is shaped around old Bannerlord-specific fields such as:

* `suggestedBannerlordAction`
* `bannerlordActionTaken`

Deprecated legacy records:

* must not be deleted
* must not be migrated
* must not be rewritten
* must not be adapted into clean v0 records
* must not enter LLM context as binding constraints
* must not expose Bannerlord-specific fields as core product fields
* should be counted/logged as deprecated or skipped after successful JSON parsing

Do not build a compatibility adapter that maps old Bannerlord fields into the clean v0 model.

## JSON Loading Rule

Do not change enforcement loading to lenient JSON parsing in this task unless investigation proves there is already a safe established enforcement-specific pattern.

If malformed enforcement JSON currently fails through `store.js`, preserve that behavior and report it.

Do not introduce runtime placeholder enforcement records.

Do not create a path where malformed enforcement placeholders can be written back during snapshot restore.

Count parsed-but-invalid clean v0 records as invalid/deprecated.

Do not promise malformed JSON counting unless malformed JSON can already be reached safely without changing the persistence contract.

## Active Scope Source

Implement the smallest possible active-scope source.

Preferred v0 source:

```json
{
  "activeScopes": [
    {
      "type": "actor",
      "id": "actor_dawud_al_sahir"
    },
    {
      "type": "organization",
      "id": "org_al_sahir"
    },
    {
      "type": "household",
      "id": "household_al_sahir"
    }
  ]
}
```

Read this from `campaign.state.activeScopes` if present.

Do not edit `campaign/state.json` to add `activeScopes` in this task.

The controller must support `campaign.state.activeScopes` if present.

If `campaign.state.activeScopes` is absent, use only this runtime active scope:

```json
{
  "type": "global",
  "id": "campaign"
}
```

The global scope exists only so records explicitly scoped to `{ "type": "global", "id": "campaign" }` can apply globally.

Verification of non-global matching may use temporary in-memory/controller fixtures, not committed campaign data.

Do not infer active scopes from:

* actor names
* clan strings
* faction strings
* Bannerlord culture
* current location text
* prose
* thread summaries
* message text
* legacy enforcement authority strings

Do not add an organization system.

Do not add hierarchy.

Do not decide that a realm contains a clan, a clan contains an actor, or a household belongs to an organization.

Exact matching only.

## Matching Rule

A clean Enforcement v0 record applies to LLM prose context when:

```text
record.status === "active"
AND
at least one record.scopes item exactly matches one active scope item by both type and id
```

Example match:

```json
{
  "recordScopes": [
    {
      "type": "organization",
      "id": "org_al_sahir"
    }
  ],
  "activeScopes": [
    {
      "type": "organization",
      "id": "org_al_sahir"
    }
  ]
}
```

Example non-match:

```json
{
  "recordScopes": [
    {
      "type": "realm",
      "id": "realm_khuzait"
    }
  ],
  "activeScopes": [
    {
      "type": "organization",
      "id": "org_al_sahir"
    }
  ]
}
```

This does not match in v0.

Do not build:

* realm-to-organization matching
* organization-to-actor matching
* location hierarchy
* authority hierarchy
* weighting
* scoring
* inheritance
* fuzzy matching

## Feed Projection Versus LLM Context

Do not treat `enforcementRecords` and LLM binding constraints as the same selection.

`enforcementRecords`:

* is an API/feed projection
* is intended as the future second-Chronicle-style Enforcement feed
* may include all clean v0 records regardless of active scope
* may include active, resolved, and retired clean v0 records
* must not include deprecated legacy records
* must not include malformed/invalid records
* must not expose deprecated Bannerlord fields
* must not expose unknown extra fields as core API fields

LLM binding constraints:

* are prose-control context
* must include only clean v0 records with `status: "active"`
* must include only records whose scopes exactly match the current active scope set
* must not include resolved records
* must not include retired records
* must not include nonmatching active records
* must not include deprecated legacy records

In short:

Feed projection answers:

```text
What enforceable laws, customs, obligations, and restrictions exist in the campaign record?
```

LLM context answers:

```text
What active enforceable constraints apply to the current prose scene?
```

## Implementation Shape

Create a focused controller/helper, likely:

```text
src/enforcementContextController.js
```

The exact filename may vary if the project has a clearer existing convention.

The controller should own:

* clean v0 validation
* deprecated legacy classification
* active scope extraction
* exact scope matching
* compact LLM context formatting
* clean API/feed projection

`campaignPacketBuilder` should remain the assembler/foreman. It should call the enforcement controller, not own enforcement validation and matching logic directly.

`server.js` should not become the enforcement logic owner.

## LLM Context Behavior

Matching active clean Enforcement v0 records should be inserted into `campaignPacket.context` as compact binding constraints.

Do not insert `enforcementRecords` wholesale into `campaignPacket.context`.

The LLM context must use the enforcement controller's scope-matched active constraint selection, not the feed projection.

Resolved, retired, deprecated legacy, invalid, and nonmatching active records must be excluded from LLM binding constraints even if they appear in the `enforcementRecords` feed projection.

Suggested formatting:

```text
Binding constraints:
- [law] Wergild is paid first to kin, then to battle-brothers if no kin exist. The clan takes no share.
- [custom] On the first night of each season, the household sets bread and drink for the absent and names one hope.
```

For records using `requiredAction`:

```text
Binding constraints:
- [obligation] Resolve Lord Manteos' demanded 5,000 denar fine.
```

Do not include:

* `suggestedBannerlordAction`
* `bannerlordActionTaken`
* legacy raw authority strings
* JSON blobs
* internal validation data
* skipped/deprecated legacy record contents
* unknown extra fields

Cap selected context records at a small deterministic number, preferably 5.

Use deterministic ordering. Prefer `createdAt` if safely comparable, otherwise stable `id` ordering.

Do not build relevance weighting.

Log something like:

```text
[campaignPacket] enforcement context: 2 selected, 1 inactive, 1 retired/resolved, 1 deprecated legacy, 0 invalid clean records
```

Exact wording may vary, but selected/skipped/deprecated counts should be visible.
